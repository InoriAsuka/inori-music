# Plan: Phase 4 Admin API Authentication

## Requirement Version

v0.4.0

## Goals

- Protect storage administration routes before adding persistence or real backend probes.
- Keep `/healthz` unauthenticated so process health checks continue to work in local and deployed environments.
- Make insecure admin access an explicit opt-in development mode instead of the default runtime behavior.
- Document authentication configuration and test both secure and insecure paths.

- [x] Append `v0.4.0` to `requirement.md`.

## Phase 2: HTTP Authentication Layer

- [x] Add handler options for admin bearer tokens and explicit insecure development mode.
- [x] Require bearer authentication on storage administration routes.
- [x] Keep `GET /healthz` public.
- [x] Return stable JSON errors for missing tokens, invalid tokens, and missing server auth configuration.
- [x] Use constant-time token comparison.

## Phase 3: Server Runtime Configuration

- [x] Read `INORI_ADMIN_TOKEN` from the environment.
- [x] Reject missing tokens by default and always reject weak configured tokens.
- [x] Allow `INORI_INSECURE_DEV_AUTH=1` for local development only.
- [x] Update runtime documentation and examples.

## Future Implementation Tasks

- [ ] Replace static bearer-token auth with user/session or OIDC-backed administrator authorization.
- [ ] Add audit logging for all storage administration mutations.
- [ ] Add secret encryption for stored backend configuration.
- [ ] Add OpenAPI security schemes to the API contract.
- [ ] Add PostgreSQL persistence and migrations.

## Completion Notes

This phase adds a minimal but explicit authentication guard for administrative routes. It is suitable for early 0.x development and self-hosted testing, but full role-based authorization and audit trails remain future work.
