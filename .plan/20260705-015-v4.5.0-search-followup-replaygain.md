# Phase v4.5.0 — 搜索收尾（高亮/拼音/历史）+ ReplayGain 自动分析 + 全量重建索引

**版本**: v4.5.0
**状态**: complete
**方向**: 全栈（Go 搜索接口扩展 + 后台分析任务 + CLI 工具 + Flutter 搜索体验）

## 背景

v4.3.0 把搜索后端从 PostgreSQL 全文搜索切换到 Meilisearch，但只消费了最基础的 ID 匹配能力：
`search.Service` 接口（`internal/search/interface.go`）的 `SearchResult{Artists, Albums, Tracks []string}`
只携带 ID，`MeilisearchService.Search()`（`internal/search/meilisearch.go:43`）的 `searchIdx` 闭包只提取
`hit["id"]`，丢弃了 Meilisearch 已支持的高亮片段能力（vendored `meilisearch-go@v0.36.3/types.go:632`
的 `SearchRequest` 已有 `AttributesToHighlight`/`HighlightPreTag`/`HighlightPostTag` 字段，当前完全未使用）。
中文标题搜索没有拼音匹配能力（用户输入拼音无法命中）。Meilisearch 索引只在 `catalog.Service` 写入时
增量同步（`CreateTrack`/`UpdateTrack`/`ImportTrack` 等触发 `go func() searchSvc.IndexTrack(...)`），
没有全量重建工具——如果索引损坏或 schema 变更，唯一恢复手段是逐条重新触发写入，没有 `cmd/reindex/`
这样的独立 CLI。Flutter `search_screen.dart` 没有搜索历史（`pubspec.yaml` 已含 `shared_preferences: ^2.3.0`
但未在搜索场景使用）。另外 v4.1.0 引入的 `replayGainDb` 字段（`catalog.Service.UpdateTrack` 中简单透传，
`services/api/internal/catalog/service.go`）目前只能靠客户端手动 PATCH 写入，服务端没有自动分析能力——
没有任何 ffmpeg/响度分析代码路径存在于 Go 代码库中。v4.5.0 补齐这几项搜索与音频侧的收尾工作。

## 目标

- 搜索结果支持高亮匹配片段，前端可视觉突出命中文本
- 中文标题支持拼音搜索
- 提供独立的全量重建索引 CLI 工具
- Flutter 端支持搜索历史记录
- 曲目上传/导入后自动分析 ReplayGain 响度值，无需人工介入

  - 文件：`services/api/internal/search/meilisearch.go`（`IndexTrack`/`IndexAlbum`/`IndexArtist`/`ensureIndexes`）

- [ ] **OpenAPI contract 更新**

### 服务端 — ReplayGain 自动分析（Go `services/api/`）

- [ ] **响度分析触发点**
  - `catalog.Service.ImportTrack` / `CreateTrack` 成功后，异步（`go func()`，与现有 `searchSvc` 索引同步
    模式一致）触发响度分析任务
  - 文件：`services/api/internal/catalog/service.go`（`ImportTrack`/`CreateTrack`）

- [ ] **分析实现（外部工具调用）**
  - 新增 `internal/audioanalysis` package：封装调用外部工具（ffmpeg `loudnorm` filter 或 `loudgain`）
    对已存储的媒体文件计算 EBU R128 响度，产出 `replayGainDb float64`
  - 需要先从存储后端（`storage.Service.GetObject`）取回音频文件到临时路径供外部工具读取，处理完成后清理临时文件
  - 文件：`services/api/internal/audioanalysis/analyzer.go`（新建）

- [ ] **分析结果写回**
  - 分析完成后调用现有 `catalog.Service.UpdateTrack(ctx, trackID, UpdateTrackRequest{ReplayGainDb: &value})`
    写入（复用 v4.1.0 已有的透传路径，不新增写入接口）
  - 分析失败（外部工具不存在/超时/解析失败）仅记录 log，不影响导入主流程，`replayGainDb` 保持 nil
  - 文件：`services/api/internal/catalog/service.go`（`UpdateTrack` 调用点）

- [ ] **单元测试**
  - `audioanalysis.Analyzer`：mock 外部命令执行，验证成功解析 / 命令不存在 / 输出格式异常三种路径
  - `MeilisearchService.Search` 高亮：验证 `_formatted` 提取正确性
  - `cmd/reindex`：验证分页遍历逻辑（可通过抽取纯函数 + 表驱动测试覆盖，不依赖真实 Meilisearch）
  - 文件：`services/api/internal/audioanalysis/analyzer_test.go`、`services/api/internal/search/meilisearch_test.go`

## 非目标

- 拼音搜索的模糊音/声调匹配（仅支持全拼精确匹配，不做同音字纠错）
- ReplayGain 分析的批量历史曲目回填工具（仅覆盖新导入曲目；历史曲目回填可用 v4.5.0 新增的
  `cmd/reindex` 类似模式后续单独开发，本阶段不做）
- 搜索历史跨设备同步（仅本地 `SharedPreferences`）
- 高亮片段的多字段合并展示（如同时高亮 artistName，仅覆盖 title/name 主字段）

## 后续候选

- 历史曲目 ReplayGain 批量回填 CLI（`cmd/analyze-backfill`，复用本阶段 `audioanalysis` package）
- 拼音模糊匹配（声调无关、简拼支持，如 "zj" 匹配 "周杰伦"）
- 搜索历史同步到服务端（跨设备共享，需要新增 viewer 域端点）
- Meilisearch 同义词库（如英文/中文别名映射）
