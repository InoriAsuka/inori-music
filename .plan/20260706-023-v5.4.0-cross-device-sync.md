# Phase v5.4.0 — 跨设备一致性（播放续播 + 搜索历史同步）——v5 封版收官

**版本**: v5.4.0
**状态**: planned
**方向**: 全栈（服务端 viewer 域新端点 + web/Flutter 双端消费）

## 背景

v5 主轴「产品化/对外开放」走到 v5.3.0 后，双端功能已对齐，但**状态仍是孤岛**：
播放队列/进度存在 web 的 localStorage（v5.2.0）与 mobile 的内存态，搜索历史存在
web localStorage（v5.1.0）与 mobile SharedPreferences（v4.5.0）。手机听到一半、
打开电脑无法继续——这是多端产品的标志性断点，也是 v4.5.0 / v5.1.0 plan 文件
「后续候选」中两次出现的"服务端同步"事项的正式落地。

本阶段完成后 v5 线封版：安全基线（v5.0）+ 双端体验对齐（v5.1–v5.3）+
跨设备一致（v5.4）构成完整的产品化闭环；后续新方向按 v6+ 升版。

## 目标

- 设备 A 播放中断后，设备 B 打开可提示「继续上次播放」并恢复队列与进度
- 搜索历史多端共享（本地与服务端合并去重）
- 冲突语义明确：last-write-wins（按 updatedAt），无静默丢失

## 任务清单

### 服务端 — 播放状态端点（`services/api/`）

- [ ] migration：新表 `user_player_state`（`user_id TEXT PRIMARY KEY`、`queue_track_ids JSONB`、
      `current_index INT`、`position_ms BIGINT`、`repeat_mode TEXT`、`shuffle BOOL`、
      `updated_at TIMESTAMPTZ`）；单行 upsert 模型，无历史版本
- [ ] `internal/playerstate` package：`Get`/`Put`（Put 校验 index 界内、queue 长度上限 500、
      updatedAt 由服务端时钟生成）；memory + PostgreSQL 双实现（沿用既有 repo 模式）
- [ ] HTTP：`GET /api/v1/me/player-state`（404 当从未上报）、
      `PUT /api/v1/me/player-state`（200 返回含服务端 updatedAt 的最新状态）；
      `requireViewerAuth` 保护；`/readyz` 追加 `playerstate_service` 检查
- [ ] OpenAPI：新 schema `PlayerState` + 两个 operation，契约版本升至 `5.4.0`

### 服务端 — 搜索历史端点（`services/api/`）

- [ ] migration：`user_search_history`（`user_id`、`query`、`searched_at`，
      `(user_id, query)` 唯一，upsert 刷新时间戳）
- [ ] HTTP：`GET /api/v1/me/search-history`（最近 20 条，按时间倒序）、
      `PUT /api/v1/me/search-history`（整体替换式写入，服务端裁剪到 20 条）、
      `DELETE /api/v1/me/search-history`（清空）
- [ ] 单元测试：裁剪 / 去重 / 空写入清空语义

### Web 客户端（`services/web/`）

- [ ] 播放状态上报：播放中每 30s 节流 + 切歌/暂停时立即 PUT（复用 v5.2.0
      持久化的序列化结构）；登出停止上报
- [ ] 恢复流程：启动时 GET，若服务端 `updatedAt` 晚于本地持久化状态，
      PlayerBar 顶部提示条「在另一台设备听到 <标题>，继续播放？」——确认后重建队列
      并 seek 到 position（不自动播放，等用户手势）
- [ ] 搜索历史：本地 ∪ 远端合并（去重取最新，裁到 20）后 PUT 回写；
      单删/清空同步远端

### Flutter 客户端（`services/mobile/`）

- [ ] `PlayerNotifier` 状态上报：同 web 语义（30s 节流 + 切歌/暂停触发，
      挂在既有 `currentIndexStream`/`playerStateStream` 订阅上，v4.7.0 基础）
- [ ] 启动恢复提示（对话框或 MiniPlayerBar 提示条），确认后 `playQueue` + seek
- [ ] `SearchHistoryNotifier`（v4.5.0）改造：本地 SharedPreferences 为主、
      登录后与服务端合并同步，离线时仅本地（回联后下次合并）

### 验证

- [ ] Go 单元测试 + OpenAPI contract tests + `-race` 绿
- [ ] Playwright e2e：模拟设备 A（PUT 状态）→ 刷新页面 → 断言恢复提示出现并可恢复
- [ ] flutter test：上报节流逻辑 / 合并策略单测
- [ ] 人工验收：手机播放 → 暂停 → 打开 web 出现继续提示 → 恢复位置误差 < 5s

## 非目标

- 实时多设备遥控（remote control / 播放接力推送）——需要 WebSocket/SSE，列候选
- 播放状态历史版本/多设备并存（单行 last-write-wins 足够）
- EQ/音量等音频设置的跨设备同步（设备相关性强，刻意不同步）
- 离线队列合并（mobile 离线域独立处理）

## 后续候选

- WebSocket/SSE 实时状态推送（设备间即时接力，Sonos 式遥控）
- 「我的设备」管理页（会话列表已有，可扩展设备命名）
- 收听位置记忆（有声书场景的 per-track resume，当前按队列粒度）
