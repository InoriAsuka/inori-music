# Phase v5.1.0 — Web 体验对齐 I（歌词面板 + 搜索高亮/历史）

**版本**: v5.1.0
**状态**: planned
**方向**: Web 客户端（`services/web/`，消费既有服务端能力，无服务端变更）

## 背景

v4 全部投入 Flutter 客户端后，web 播放器明显掉队：2026-07-06 审查确认
`services/web/` 下**没有任何歌词相关代码**（grep lyrics 零命中），搜索页
（`app/(app)/search/page.tsx`）没有高亮渲染、没有搜索历史。而服务端能力已全部就绪：

- 歌词：`GET /api/v1/catalog/tracks/{id}/lyrics`（v4.0.0）+ 翻译/来源字段（v4.4.0）
- 搜索高亮：`SearchResultItem.highlight`（`<mark>` 片段，v4.5.0；PG 降级时为空）
- 拼音搜索：服务端索引侧实现（v4.5.0），web 端零成本受益

本阶段是纯前端消费，参照 Flutter 端已验证的交互模式移植，是 v5 产品化主轴中
「双端一致体验」的第一步。与 v5.0.0 无依赖，可并行。

## 目标

- Web 播放页有同步滚动歌词，逐字高亮（有 word-level 数据时）与双语翻译对齐 Flutter 端
- Web 搜索结果高亮命中片段，且渲染路径无 XSS 风险
- Web 搜索有本地历史（行为对齐 Flutter 端：最近 20 条、去重、单删/清空）

## 任务清单

### 歌词面板（`services/web/`）

- [ ] `lib/lyrics/lrcParser.ts`：移植 `services/mobile/lib/src/lyrics/lrc_parser.dart` 的
      解析逻辑——行级 `[mm:ss.xx]` 时间戳、行内 `<mm:ss.xx>` 逐字 spans、翻译文本配对；
      输出与 mobile `LyricLine`（`words` + `translation`）等价的 TS 类型
- [ ] `components/player/LyricsPanel.tsx`：
      从 PlayerBar 展开（或全屏播放视图内）；随 `useAudio` position 同步滚动；
      当前行高亮，有 `words` 数据时逐字渐变（CSS `background-clip: text` 渐变或分 span 着色），
      无逐字数据回退整行高亮；当前行下方渲染次要样式翻译文本
- [ ] 双语开关与「无歌词」空状态；设置持久化 localStorage
- [ ] 歌词数据获取：track 变化时 GET lyrics 端点，404 静默处理，结果按 trackId 缓存

### 搜索高亮 + 历史（`services/web/app/(app)/search/page.tsx`）

- [ ] 高亮渲染：解析后端返回的 `highlight` 字段（`<mark>...</mark>` 简单标签），
      **以字符串切分方式转为 React 元素**（`<mark>` 内文本用 primary 色 + 加粗），
      **禁止 `dangerouslySetInnerHTML`**——高亮片段源自用户可控的曲目标题，必须防注入；
      `highlight` 为空（PG 降级）时回退纯文本
- [ ] `lib/searchHistory.ts` + hook：localStorage 持久化最近 20 条查询（去重、最新在前），
      输入框聚焦且为空时下拉展示，支持单条删除与清空全部（对齐 mobile
      `search_history_provider.dart` 行为）
- [ ] 键盘可达性：历史下拉支持 ↑↓ 选择 + Enter 提交

### 验证

- [ ] Playwright e2e：搜索关键词 → 断言 `<mark>` 高亮元素存在；再次聚焦 → 断言历史条目出现；
      播放有歌词曲目 → 断言歌词面板行随时间推进高亮
- [ ] Vitest：`lrcParser.ts` 表驱动测试（行级/逐字/翻译/畸形输入），用例数据对齐
      mobile 端 `lrc_parser` 既有测试
- [ ] `npm run type-check` + `npm run lint` 通过
