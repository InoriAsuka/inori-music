## 本阶段发现的缺陷（2026-07-08 ~ 2026-07-10 会话期间）

在补齐测试基建、驱动真实播放/收藏/历史/登出流程的过程中，发现以下此前未被发现的真实缺陷
（部分是纯测试基建问题，已在描述中标注）：

1. bcrypt 相关测试在 `-race` 下超时（已修复，测试侧加大 cost 参数容忍度/调整超时）
2. CI 用户名含连字符导致 workflow 语法问题（已修复）
3. CSS `@import` 顺序问题（已修复）
4. `AuthProvider` 挂载位置不当（已修复）
5. 重复的 `/` 路由定义（已修复）
6. 错误的相对 CSS import 路径（已修复）
7. auth service 为 nil 时未判空导致 panic（已修复）
8. **`ReplayGainDb` 静音音轨写入 `+Inf`**（`services/api/internal/audioanalysis/analyzer.go`）：
   ffmpeg loudnorm 对绝对静音报告 `-inf` LUFS，`referenceLoudnessLUFS - (-Inf) = +Inf`，
   `encoding/json` 无法序列化非有限浮点数，导致该曲目此后所有 GET 请求返回 200 但空 body。
   修复：`math.IsNaN`/`math.IsInf` 判空后跳过 writeback（见 `analyzer.go:111-114`，
   `analyzer_test.go` 的 `TestAnalyzerSkipsWritebackOnSilentTrack` 覆盖）
9. **收藏页永久显示"空"**（`services/web/app/(app)/library/favorites/page.tsx`）：
   `GET /api/v1/me/favorites/tracks` 在 catalog service 可用时（生产环境默认路径）只返回
   `tracks` 数组，不返回 `trackIds`；页面的空态判断只检查了 `trackIds.length === 0`，
   导致收藏功能界面上"看起来"完全不工作。修复：空态判断同时检查 `tracks.length === 0`
10. **Playwright 专辑计数竞态**（测试基建问题，非产品 bug）：`main-flow.spec.ts` 中
    `page.goto("/albums")` 后立即 `.count()` 会与客户端数据请求竞态，读到 0 导致测试静默
    自我跳过。修复：`.count()` 前加 `waitFor({state:"visible"})` 保护
11. **播放历史从未被真正记录**（`services/web/hooks/useAudio.ts`，本会话发现的最严重缺陷）：
    `onEnded` 处理器 POST `/api/v1/me/history` 时携带了 API 拒绝的多余字段
    （`durationSeconds`、`source`），Go 端严格 JSON 解码返回 400，但错误被
    `.catch(() => {})` 静默吞掉——用户界面无任何异常表现，但生产环境的播放历史记录
    功能实际上从未成功写入过一条记录。修复：POST body 仅保留 API 契约认可的
    `trackId`/`playedAt` 两个字段
12. **登出点击后不跳转 `/login`**（`services/web/components/layout/AuthProvider.tsx`）：
    该组件的 zustand store 订阅回调在监测到 `token` 变为空时会调用 `clearSession()`，
    而 `clearSession()` 本身触发 `set()` 又会同步重新通知同一个订阅者，形成无限递归直至
    JS 调用栈溢出；异常在 `Topbar.tsx` 的 `handleLogout()` 内部（`clearSession()` 调用处）
    抛出，导致其后的 `router.push("/login")` 永远不会执行——点击"登出"按钮时 POST 请求
    正常发出，但页面停留在原地。修复：订阅回调中移除多余的 `clearSession()` 调用，
    只保留 cookie 清理这一必要副作用
