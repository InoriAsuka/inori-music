# Phase v4.8.1 — v4.8.0 推送后 CI 发现的 3 个真实缺陷修复

**版本**: v4.8.1
**状态**: 已完成（2026-07-11 封版）
**方向**: v4 封版后的 bug 修复 patch（按 v4.8.0 计划文件约定，v4.x 不再规划新 Phase）

## 背景

v4.8.0 打 tag 推送后，检查远端 GitHub Actions 发现最近 5 次 CI 运行全部失败——
与本地 pre-push 验证（gofmt/go test -race/biome lint/flutter analyze/flutter test/
vitest/playwright 全绿）矛盾。逐条拉取失败日志（`gh run view <id> --log-failed`）
定位到 3 个独立根因，均与 v4.8.0 阶段本身的功能代码无关（1 个是本阶段测试代码引入的
真实疏漏，另外 2 个是长期存在、此前从未被 CI 触达或从未被本仓库改动触发的 pre-existing
缺陷）。

## 任务清单

### 1. Flutter CI：`make gen:api` 触发 Makefile 静态模式规则解析错误

- **现象**：`Makefile:1: *** target pattern contains no '%'.  Stop.`，报错行号指向
  `.PHONY` 声明行而非实际的 `gen:api:` 规则定义行，具有误导性。
- **根因**：`services/mobile/Makefile` 的目标名 `gen:api`/`build:watch` 含冒号。GNU Make
  对含 2 个以上冒号的行按静态模式规则语法解析（`targets: target-pattern: prereqs`），
  而非当作字面含冒号的目标名；`api`/`watch` 都不含规则要求的 `%` 通配符，Make 因此在
  文件第一行含 2+ 冒号处（即 `.PHONY` 行）就直接 abort。
- **溯源**：确认是 commit `e314b83`（Phase 304-308）引入的既有缺陷，此前一直未暴露——
  CI 的 Flutter job 从未跑到 `make gen:api` 这一步，被本会话更早修复的另一个问题
  （`flutter-version` 钉死 `3.27.x` 与 mobile 应用 Dart SDK floor `^3.12.2` 不匹配导致
  `flutter pub get` 失败）挡在前面。修完那个问题后，这个问题才第一次真正跑到并暴露。
- **修复**：`services/mobile/Makefile` 内两个目标改用连字符（`gen-api`、`build-watch`），
  同步更新 `.github/workflows/flutter.yml` 5 处调用与 `services/mobile/README.md` 1 处调用。
- **验证**：本地 `make gen-api` 端到端跑通；`git status --short` 确认重新生成的 Dart
  API client 与已提交内容零 diff（确定性代码生成，修复不产生副作用）。

### 2. Web CI：`e2e/main-flow.spec.ts` 冗余 `@ts-expect-error` 导致 type-check 失败

- **现象**：`error TS2578: Unused '@ts-expect-error' directive.`
  （`e2e/main-flow.spec.ts(60,5)`）。
- **根因**：`window.Audio = ProbedAudio` 一行的 `@ts-expect-error` 抑制的并非真实类型错误——
  `class ProbedAudio extends NativeAudio` 结构上可赋值回 `window.Audio` 原类型，TS strict
  模式下会将"未产生任何错误的抑制注释"本身判定为错误。
- **本阶段引入方式的自我复盘**：这是 v4.8.0 本阶段新增测试代码（`main-flow.spec.ts`）
  自身携带的真实疏漏，而非既有代码缺陷。push 前跑过 `npx playwright test`（通过）、
  `npx vitest run`（49/49 通过）、`biome lint`（0 errors），唯独没有单独针对这个新增
  e2e 文件跑一次 `npm run type-check`，导致这个编译期错误未被本会话自己的验证流程拦下。
  记录在此，作为后续会话的检查清单补充：新增测试/spec 文件后，`type-check` 必须与
  test runner 本身分开单独确认一次，不能假设"测试跑过 = 类型也对"。
- **修复**：移除该行多余的 `@ts-expect-error`（保留上一行 `window.__testAudioInstance = this`
  的注释，那一处是真实需要抑制的 untyped 全局属性赋值）。
- **验证**：`npm run type-check`（services/web）修复前复现报错，修复后 0 errors。

### 3. Docker 镜像发布 CI：`services/web`/`services/admin` 镜像构建失败

- **现象**：`buildx failed ... "/packages/api-contract": not found`，`publish-web`/
  `publish-admin` 两个 job 失败，`publish-api` 正常。
- **根因 A（实际触发失败的原因）**：仓库根 `.dockerignore` 含裸 `packages` 一行，导致
  `packages/` 整个目录（含 `packages/api-contract/` 与 `packages/ui/`）被排除在所有以
  `.` 为 context 的 Docker 构建之外——BuildKit 从未接收到这些文件，两个 Dockerfile 里的
  `COPY packages/api-contract ../../packages/api-contract` 因此必然失败。
- **根因 B（此前被 A 掩盖、A 修复后会立即暴露的第二个缺陷）**：`services/web/package.json`
  与 `services/admin/package.json` 均声明 `"@inori/ui": "file:../../packages/ui"`；
  `services/admin/components/admin/StorageHealthBadge.tsx` 有真实的
  `export { StorageHealthBadge } from "@inori/ui"` 引用；两个 `next.config.ts` 均声明
  `transpilePackages: ["@inori/ui"]`。但两个 Dockerfile 都完全没有任何
  `COPY packages/ui ...` 指令——本地用等价目录结构复现验证：`npm ci` 在 `file:` 目标
  完全缺失时仍返回 exit 0（只是留下一个悬空 symlink），真正的失败点在后续
  `next build`/任何 `require`/`import` 实际解析该符号链接时（`Cannot find module
  '@inori/ui'`），也就是两个 Dockerfile 的 `RUN npm run build` 步骤。
- **回归窗口溯源**：用 `gh run list --workflow=Docker --json conclusion,createdAt,
  databaseId,headSha` 逐次对比，确认该 Docker workflow 自 commit `4b184bc8`
  （2026-06-21，v2.2.0 阶段）起连续失败超过 26 次、跨越约 19 天，一路贯穿 v2.2.0 到
  v4.8.0 全部发布——`git diff` 确认 `.dockerignore`/两个 Dockerfile 在最后一次成功
  （`940d1198`）与第一次失败（`4b184bc8`）之间的内容**完全没有变化**，判断触发因素
  是 GitHub 托管 runner 的 BuildKit/buildx 版本升级改变了"COPY 源缺失"的处理方式
  （从静默/宽松变为硬错误），而非本仓库自身改动。**两个缺陷均早于 v4 整条线存在，
  与 v4.8.0 本阶段工作内容无关**，只是恰好在本次逐条排查 CI 时第一次被发现并修复。
- **修复**：
  - `.dockerignore` 移除 `packages` 排除行。
  - `services/web/Dockerfile`、`services/admin/Dockerfile` 的 `builder` stage 各追加
    一行 `COPY packages/ui ../../packages/ui`（紧跟已有的 `COPY packages/api-contract
    ../../packages/api-contract` 之后，`RUN npm run build` 之前）。
- **验证局限**：本地无 Docker 环境（`which docker` 无结果），无法实际跑一次
  `docker build` 验证。修复依据：(1) 用等价目录深度在本地复现了 npm `file:` symlink
  的解析行为（`node_modules/@inori/ui -> ../../../../packages/ui`，相对路径层级与
  Dockerfile 内 `WORKDIR /app` + `COPY ... ../../packages/ui` 的目标路径一致）；
  (2) 确认 `packages/ui`、`packages/api-contract` 目录下均无 `.dockerignore` 会误伤的
  `*.md`/`node_modules`/`dist` 内容。真正的确认只能等下一次推送触发的实际 CI 运行。

## 非目标

- 不改动 v4.8.0 已锁定的功能代码/测试断言本身（这 3 个修复全部是构建/CI 配置层面）
- 不因这几个 pre-existing 缺陷的发现而重新打开 v4.8.0 阶段文件（遵循"phase 一旦
  封版即冻结"的既有约定，另起本文件记录）
- 不同步 OpenAPI contract 版本号（本 patch 无 API schema 变更）

## 后续候选

- 若未来 GitHub 托管 runner 的 buildx 行为再次变化，可考虑在 CI 里显式钉住
  `docker/setup-buildx-action` 的版本，避免同类"无本仓库改动却因上游依赖升级而
  回归"的情况再次发生且需要事后排查才能定位
- 本次为定位 Docker 回归窗口用到的"二分 CI 运行历史"方法（`gh run list --json` +
  逐次 `git diff` 核对）可考虑沉淀为一个可复用的排查脚本
