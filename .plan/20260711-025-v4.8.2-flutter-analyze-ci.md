# Phase v4.8.2 — v4.8.1 推送后 CI 发现的第 4 个真实缺陷修复

**版本**: v4.8.2
**状态**: 已完成（2026-07-11 封版）
**方向**: v4 封版后的 bug 修复 patch（延续 v4.8.1 的同一批 CI 排查）

## 背景

v4.8.1 打 tag 推送后监控远端 CI，`Flutter CI` workflow 的 `Analyze` job 在 ubuntu-latest
上失败（`Process completed with exit code 1`），macOS 侧同一 job 因矩阵策略 fail-fast
被联带取消（非独立失败，同一根因）。

## 任务清单

### 4. Flutter CI：`.github/workflows/flutter.yml` 的 Analyze 步骤使用裸 `flutter analyze`

- **现象**：`flutter analyze` 报告 4 条 info 级问题（`full_player_screen.dart` 两处、
  `settings_screen.dart` 两处，均为字符串插值多余花括号，`unnecessary_brace_in_string_interps`），
  退出码 1，job 判定失败。
- **根因**：`flutter analyze`（不带任何 flag）对 info 级问题也返回非零退出码。这 4 条 info
  级问题本身早已存在且被认定为可接受（v4.8.0 计划文件验证章节明确写有"flutter analyze
  --no-fatal-infos 仅 4 条 info 级（与 CI 一致，非 fatal）"），`.github/workflows/build.yml`
  的等价 Analyze 步骤（第 109 行）也早已正确使用 `flutter analyze --no-fatal-infos`——
  但 `.github/workflows/flutter.yml` 的 Analyze 步骤（第 42 行）一直是裸 `flutter analyze`，
  两个 workflow 文件对同一命令的 flag 不一致。
- **溯源**：确认这一行自引入（commit `68411e7`，v3.5.0，新增 iOS CI build job 时一并加入
  Flutter CI workflow 的 Analyze 步骤）起就是裸命令，此前从未真正暴露——同样是被 v4.8.1
  才修复的 Flutter SDK 版本钉死问题挡在前面（`flutter pub get` 先失败，Analyze 步骤从未
  跑到）。属于与 Makefile 冒号问题、Docker COPY 问题同一批"被同一个上游 SDK 版本钉死问题
  长期掩盖的既有缺陷"，只是这一个要等 v4.8.1 的 Makefile 修复生效、`gen-api`/`build_runner`
  两步都跑过之后才轮到 Analyze 步骤执行，因此没能在 v4.8.1 那一轮排查里和前 3 个一起发现。
- **修复**：`.github/workflows/flutter.yml` 第 42 行 `flutter analyze` 改为
  `flutter analyze --no-fatal-infos`，与 `build.yml` 保持一致。
- **验证**：本地 `flutter analyze --no-fatal-infos`（services/mobile）退出码 0，
  4 条 info 级问题原样保留输出但不再判定为失败，与 `build.yml` 现有行为一致。

## 非目标

- 不修复这 4 条 info 级 lint 本身（字符串插值多余花括号是纯风格问题，不影响行为，
  且两个 workflow 文件此前的既有基线就是接受它们存在）——若后续要清零，作为独立的
  纯格式化 patch 处理，不与 CI 配置修复混在一起

## 后续候选

- 无（本 patch 范围极小，一行 flag 对齐）
